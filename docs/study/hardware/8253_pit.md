# 8253/8254 — Programmable Interval Timer (PIT)

**Source:** `kernel/src/kernel/drivers/timer.rs`

---

## What it is

The Intel 8253/8254 PIT is a programmable counter chip with three independent
16-bit channels. Each channel counts down from a programmed value at a fixed
input frequency of **1,193,182 Hz** (≈ 1.19 MHz), then fires an output signal.

**Why 1,193,182 Hz?** Historical accident. The original IBM PC ran at 14.31818 MHz
(NTSC color subcarrier frequency). That was divided by 3 for the CPU clock, then by 4
for the PIT — giving 14,318,180 / 12 ≈ 1,193,182 Hz. This oddly specific number is
now baked into every x86 PC as an immutable fact of life.

The three channels:
- **Channel 0** — connected to IRQ0. OxideOS uses this as the system timer.
- **Channel 1** — historically used for DRAM refresh. Not used in modern systems.
- **Channel 2** — connected to the PC speaker. Can generate audio tones.

---

## I/O Ports

| Port | Name | Direction | Purpose |
|------|------|-----------|---------|
| `0x40` | Channel 0 | Read/Write | Counter 0 data (low byte then high byte) |
| `0x41` | Channel 1 | Read/Write | Counter 1 data (unused by OxideOS) |
| `0x42` | Channel 2 | Read/Write | Counter 2 data (PC speaker) |
| `0x43` | Mode/Command | Write only | Configure any channel |

---

## The Mode/Command Register (port 0x43)

This 8-bit register selects the channel and its operating mode.

```
Bits [7:6] — Channel select
  00 = Channel 0
  01 = Channel 1
  10 = Channel 2
  11 = Read-back command (not used by OxideOS)

Bits [5:4] — Access mode
  00 = Latch count value for reading
  01 = Low byte only
  10 = High byte only
  11 = Lo/Hi byte (send low byte first, then high byte)

Bits [3:1] — Operating mode
  000 = Mode 0: Interrupt on terminal count
  001 = Mode 1: Hardware re-triggerable one-shot
  010 = Mode 2: Rate generator (periodic IRQ) ← used by OxideOS
  011 = Mode 3: Square wave generator (PC speaker)
  100 = Mode 4: Software-triggered strobe
  101 = Mode 5: Hardware-triggered strobe

Bit [0] — BCD/Binary
  0 = Binary (16-bit) ← used by OxideOS
  1 = BCD (4 digits)
```

**OxideOS uses `0x34`** for Channel 0:
```
0x34 = 0b 00 11 010 0
           ││ ││ │││ └── Binary mode
           ││ ││ └┴┴─── Mode 2 (Rate Generator)
           ││ └┴──────── Lo/Hi byte access
           └┴──────────── Channel 0
```

## How frequency is programmed

The PIT divides its 1,193,182 Hz input by the **divisor** you write.

```
divisor = 1,193,182 / desired_frequency_hz
```

For OxideOS's 100 Hz timer (`timer.rs` line 8–9):
```rust
let pit_freq = 1_193_182;
let divisor = pit_freq / freq_hz;  // 1,193,182 / 100 = 11,931 = 0x2E9B
```

The divisor is written as two bytes (low then high) to port `0x40`:
```rust
asm!("out dx, al", in("dx") 0x43u16, in("al") 0x34u8, ...); // mode command
asm!("out dx, al", in("dx") 0x40u16, in("al") divisor_low, ...);  // low byte
asm!("out dx, al", in("dx") 0x40u16, in("al") divisor_high, ...); // high byte
```

After this, Channel 0 fires **IRQ0 every 10ms (100 times per second)**.

Special case: divisor `0` means 65536, giving the lowest possible frequency ≈ 18.2 Hz
(the original PC-BIOS clock rate).

---

## Mode 2: Rate Generator

In Mode 2, the channel:
1. Loads the divisor
2. Counts down to 1 (not 0)
3. Pulses the output LOW for one cycle
4. Automatically reloads the divisor and repeats

This generates a periodic interrupt at the programmed rate. The output pulse
triggers IRQ0 → the timer ISR runs → kernel increments `TIMER_TICKS` → scheduler
gets a chance to preempt the current task.

---

## Tick counter and time

In `kernel/src/kernel/arch/interrupts.rs`, the timer ISR increments a global:
```rust
pub static mut TIMER_TICKS: u64 = 0;
```

Time is derived from this in `timer.rs`:
```rust
pub unsafe fn get_uptime_ms() -> u64 {
    TIMER_TICKS * 10       // 100 Hz = 10ms per tick
}
pub unsafe fn get_uptime_seconds() -> u64 {
    TIMER_TICKS / 100      // 100 ticks = 1 second
}
```

---

## RDTSC — High-resolution timer

For finer timing, OxideOS also has `rdtsc()` in `timer.rs`:
```rust
asm!("rdtsc", out("eax") low, out("edx") high, ...);
```

`RDTSC` (Read Time-Stamp Counter) returns the number of CPU cycles since reset.
On a 3 GHz CPU, this increments 3 billion times per second — nanosecond precision
vs. the PIT's millisecond precision. However, it's not used for the OS scheduler
because it's CPU-frequency-dependent and can be affected by power states.

---

## Channel 2 — PC Speaker

Channel 2 is wired to the PC speaker through **port 0x61** (system control port B):
- Bit 0: enable PIT channel 2 output gate
- Bit 1: enable speaker output

To play a tone:
1. Program Channel 2 in Mode 3 (square wave): write `0xB6` to port `0x43`
2. Write the frequency divisor to port `0x42` (low then high byte)
3. Set bits 0 and 1 of port `0x61`

To stop:
- Clear bits 0 and 1 of port `0x61`

OxideOS does not currently have a speaker driver, but the hardware is there.
(See `07_drivers.md` — PC Speaker is the recommended first driver exercise.)

| `timer::sleep_ticks(n)` | `timer.rs:79` | Busy-wait for n ticks |
| `timer::rdtsc()` | `timer.rs:88` | CPU cycle counter |

After `init()`, `pic::unmask_irq(0)` must be called to enable the IRQ0 line,
otherwise the timer fires but the CPU never sees it.

## Common gotchas

**1. Order of low/high byte matters.**
You must write the low byte first, then the high byte. Writing them in the wrong
order gives you a completely wrong frequency.

**2. The command byte must come before the counter bytes.**
The mode register tells the chip how many bytes to expect. Write `0x43` first,
then write two bytes to `0x40`.

**3. Divisor 0 means 65536, not 0.**
The hardware treats divisor 0 as the maximum (65536), not division by zero.

**4. Integer division truncates.**
`1,193,182 / 100 = 11,931` (not 11,931.82). The actual timer frequency is
`1,193,182 / 11,931 ≈ 99.997 Hz`, not exactly 100 Hz. Over 24 hours this
accumulates to a drift of about 1 second. Real kernels compensate using
the hardware RTC.
