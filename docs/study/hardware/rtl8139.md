# RTL8139 — Fast Ethernet NIC

**Source:** `kernel/src/kernel/drivers/net/rtl8139.rs`

---

## What it is

The **Realtek RTL8139** is a 100 Mbit/s Ethernet controller. It was one of the
most common consumer NICs of the late 1990s and is emulated by QEMU.
OxideOS uses it with: `-netdev user,id=net0 -device rtl8139,netdev=net0`.

Compared to the PIC or PIT, the RTL8139 is a complex device:
- It has its own memory-mapped register set
- It uses DMA (hardware writes packets directly into RAM)
- It manages two ring buffers (one for receive, one for transmit)

---

## Physical Resource Discovery

The RTL8139 is a PCI device. Before using it, OxideOS must:
1. Find it on the PCI bus (`pci::scan(0x10EC, 0x8139)`)
2. Get its I/O base address from BAR0 (`device.io_bar(0)`)
3. Enable bus mastering and I/O space in the PCI command register

After this, the RTL8139's registers are accessible at `io_base + offset`.

---

## I/O Register Map

All registers are accessed via port I/O relative to `io_base` (the value from BAR0):

| Offset | Name | Size | Purpose |
|--------|------|------|---------|
| `0x00` | `REG_MAC` | 6 bytes | Hardware MAC address |
| `0x08` | `REG_MAR` | 8 bytes | Multicast address filter |
| `0x10–0x1C` | `REG_TSD[0–3]` | 32-bit × 4 | TX status descriptors |
| `0x20–0x2C` | `REG_TSAD[0–3]` | 32-bit × 4 | TX start address descriptors |
| `0x30` | `REG_RBSTART` | 32-bit | Physical address of RX ring buffer |
| `0x37` | `REG_CR` | 8-bit | Command register (enable RX/TX, reset) |
| `0x38` | `REG_CAPR` | 16-bit | Current address of packet read (our pointer into RX ring) |
| `0x3A` | `REG_CBR` | 16-bit | Current buffer address (chip's pointer into RX ring) |
| `0x3C` | `REG_IMR` | 16-bit | Interrupt mask register |
| `0x3E` | `REG_ISR` | 16-bit | Interrupt status register |
| `0x40` | `REG_TCR` | 32-bit | TX configuration register |
| `0x44` | `REG_RCR` | 32-bit | RX configuration register |
| `0x52` | `REG_CONFIG1` | 8-bit | Configuration register 1 (power management) |

## Memory Layout

The RTL8139 uses DMA — the chip writes received packets directly into RAM and
reads transmitted packets directly from RAM. The CPU sets up the addresses once;
the hardware handles the rest.

### RX Ring Buffer (receive)

An 8 KB ring buffer allocated in kernel memory. The chip writes incoming packets
here continuously. The driver tracks where to read next via `rx_offset`:

```
┌──────────────────────────────────────────────────┐
│  [pkt header][packet data][pkt header][packet..] │ 8 KB ring
└──────────────────────────────────────────────────┘
     ▲                           ▲
  rx_offset (our read pos)    CBR (chip's write pos)
```

Each packet in the ring is preceded by a 4-byte header:
- Bits 0–15: receive status (ROK = OK, errors)
- Bits 16–31: packet length (including header)

### TX Buffers (transmit)

Four static 1536-byte buffers (one per TX descriptor slot). OxideOS cycles
through them round-robin. To send a packet:
1. Copy packet data into `tx_buffers[tx_slot]`
2. Write the physical address of the buffer to `REG_TSAD[tx_slot]`
3. Write the packet length to `REG_TSD[tx_slot]` — this starts the DMA

## Initialization sequence

```rust
// 1. Power on (some cards need this)
outb(io + REG_CONFIG1, 0x00);

// 2. Software reset — clears all buffers and state
outb(io + REG_CR, CR_RESET);
while inb(io + REG_CR) & CR_RESET != 0 {}  // wait for reset complete

// 3. Tell the card where the RX ring is in physical memory
outl(io + REG_RBSTART, rx_ring_phys_addr);

// 4. Set interrupt mask — which events generate IRQs
outw(io + REG_IMR, ISR_ROK | ISR_TOK | ISR_TER | ISR_RER);

// 5. Configure the receiver
// RCR_APM = accept our MAC, RCR_AB = accept broadcast,
// RCR_AM = accept multicast, RCR_WRAP = don't stop at ring end,
// RCR_RBLEN_8K = 8KB ring size
outl(io + REG_RCR, RCR_APM | RCR_AB | RCR_AM | RCR_WRAP | RCR_RBLEN_8K);

// 6. Enable receiver and transmitter
outb(io + REG_CR, CR_RE | CR_TE);

// 7. Configure transmitter (standard settings)
outl(io + REG_TCR, 0x03000700);
```

## Receiving a packet

The driver polls `REG_ISR` for `ISR_ROK` (RX OK) each frame:

```rust
let isr = inw(io + REG_ISR);
outw(io + REG_ISR, isr);         // write back to clear the flags

if isr & ISR_ROK != 0 {
    // CBR != CAPR means there's a packet waiting
    while inw(io + REG_CBR) != rx_offset as u16 {
        // Read 4-byte packet header from ring
        let header: u32 = read_from_ring(rx_offset);
        let pkt_len = (header >> 16) as usize - 4; // minus CRC

        if header & 0x0001 != 0 {  // ROK bit in packet header
            // Copy pkt_len bytes from ring starting at rx_offset + 4
            let packet = &rx_ring[rx_offset + 4..rx_offset + 4 + pkt_len];
            // hand off to network stack...
        }

        // Advance CAPR (our read position)
        rx_offset = (rx_offset + pkt_len + 4 + 3) & !3; // align to 4 bytes
        outw(io + REG_CAPR, rx_offset as u16 - 0x10);   // tell chip where we read to
    }
}
```

---

## Interrupt Status Register (ISR) bits

| Bit | Name | Meaning |
|-----|------|---------|
| 0 | `ISR_ROK` | Receive OK — a packet arrived |
| 1 | `ISR_RER` | Receive error |
| 2 | `ISR_TOK` | Transmit OK — last TX completed |
| 3 | `ISR_TER` | Transmit error |

OxideOS writes these bits back to ISR immediately to clear them:
```rust
outw(io + REG_ISR, isr);  // writing the set bits clears them
```

---

## TX status descriptor (TSD) bits

After writing a packet address + length to TSAD/TSD, the chip sets bits in TSD:

| Bit | Name | Meaning |
|-----|------|---------|
| 13 | `TSD_OWN` | DMA complete — we own the buffer again |
| 15 | `TSD_TOK` | Transmit OK |
| 30 | `TSD_TABT` | Transmit aborted (collision limit exceeded) |

## Common gotchas

**1. Physical vs virtual addresses for DMA.**
The chip works with physical RAM addresses, not virtual addresses. OxideOS
must pass the physical address of the RX ring to `REG_RBSTART`. Using a virtual
address here would make the chip write to a wrong (possibly non-existent) location.

**2. The 0x10 offset in CAPR.**
`REG_CAPR` has a hardware quirk — you must subtract 0x10 from the actual ring
offset when writing to it. `outw(REG_CAPR, rx_offset - 0x10)`. The datasheet
documents this but it's easy to miss.

**3. Waiting for reset before configuring.**
After writing `CR_RESET`, the chip needs time to clear its internal state.
If you write `REG_RBSTART` before the reset completes, the write is lost.

**4. ISR must be cleared by writing 1s.**
This chip uses **write-1-to-clear** (W1C) semantics. Writing `0xFFFF` to ISR
clears all interrupt flags. Writing 0 does nothing.
