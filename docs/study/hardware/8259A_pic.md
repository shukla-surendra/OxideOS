# 8259A — Programmable Interrupt Controller (PIC)

**Source:** `kernel/src/kernel/drivers/pic.rs`

---

## What it is

The Intel 8259A is the chip that lets hardware devices interrupt the CPU.
Without it, the CPU would have to constantly poll every device to see if
something happened — wasting 100% of CPU time checking "did the keyboard
press a key? Did the timer tick? Did the disk finish?" Instead, devices
signal the PIC, the PIC signals the CPU, and the CPU drops what it's doing
and handles the event.

A PC has **two cascaded 8259As**:
- **Master PIC** — handles IRQ0 through IRQ7
- **Slave PIC** — handles IRQ8 through IRQ15, connected to the master via IRQ2

This gives 15 usable interrupt lines (IRQ2 is used for the cascade, not
available for devices).

```
         ┌────────────────────┐        ┌────────────────────┐
IRQ0 ────►                    │        │                    ◄──── IRQ8  (RTC)
IRQ1 ────► Master 8259A       │        │  Slave 8259A       ◄──── IRQ9
IRQ2 ────► (cascade input) ──►├──IRQ2──►                    ◄──── IRQ10
IRQ3 ────►                    │        │                    ◄──── IRQ11
IRQ4 ────►                    │        │                    ◄──── IRQ12 (Mouse)
IRQ5 ────►                    │        │                    ◄──── IRQ13
IRQ6 ────►                    │        │                    ◄──── IRQ14 (ATA primary)
IRQ7 ────►                    │        │                    ◄──── IRQ15 (ATA secondary)
         └────────┬───────────┘        └────────────────────┘
                  │ INT pin
                  ▼
               x86-64 CPU
```

## I/O Ports

| Port | Name | Direction | Purpose |
|------|------|-----------|---------|
| `0x20` | `PIC1_COMMAND` | Write | Send commands to master PIC (ICW1, OCW2/3) |
| `0x21` | `PIC1_DATA` | Read/Write | Read/write master PIC interrupt mask; send ICW2/3/4 during init |
| `0xA0` | `PIC2_COMMAND` | Write | Send commands to slave PIC |
| `0xA1` | `PIC2_DATA` | Read/Write | Read/write slave PIC interrupt mask; send ICW2/3/4 during init |

**I/O delay port `0x80`:** Writing any value here takes ~1 µs — used between PIC
writes to let the slow 8259A hardware settle. This is `io_wait()` in the code.

## The IRQ → CPU Vector mapping problem

The x86 CPU reserves interrupt vectors **0–31** for CPU exceptions:
- Vector 0: Divide by zero
- Vector 8: Double fault
- Vector 14: Page fault
- etc.

After a PC reset, the 8259A maps IRQ0 to vector 8 by default — a direct
collision with the "Double Fault" CPU exception. Any timer tick would be
misinterpreted as a double fault.

**The fix: remapping.** During `init()`, we reprogram the PIC to use vectors
32–47 (0x20–0x2F) instead:

```
IRQ0  (Timer)    → CPU vector 32  (0x20)
IRQ1  (Keyboard) → CPU vector 33  (0x21)
IRQ2  (Cascade)  → CPU vector 34  (0x22)
  ...
IRQ7             → CPU vector 39  (0x27)
IRQ8  (RTC)      → CPU vector 40  (0x28)
  ...
IRQ12 (Mouse)    → CPU vector 44  (0x2C)
IRQ14 (ATA pri)  → CPU vector 46  (0x2E)
IRQ15 (ATA sec)  → CPU vector 47  (0x2F)
```

## Initialization sequence (ICWs)

The 8259A is initialized with a sequence of **Initialization Command Words (ICWs)**.
You must write them in order — the chip expects exactly ICW1, then ICW2, ICW3, ICW4.

### ICW1 — Start initialization
Written to the **command port** (`0x20` / `0xA0`).

```
Value: 0x11
  bit 0 (ICW4 needed): 1  — we will send ICW4
  bit 1 (single):      0  — cascade mode (two PICs)
  bit 3 (level/edge):  0  — edge-triggered
  bit 4 (init):        1  — must be 1 to trigger init sequence
```

In OxideOS (`pic.rs` line 48–64):
```rust
asm!("out dx, al", in("dx") PIC1_COMMAND, in("al") 0x11u8, ...);
asm!("out dx, al", in("dx") PIC2_COMMAND, in("al") 0x11u8, ...);
```

### ICW2 — Set vector offset
Written to the **data port** (`0x21` / `0xA1`).

```
Master: 0x20  → IRQ0 maps to vector 32
Slave:  0x28  → IRQ8 maps to vector 40
```

In OxideOS (`pic.rs` line 66–83):
```rust
asm!("out dx, al", in("dx") PIC1_DATA, in("al") 0x20u8, ...); // master offset = 32
asm!("out dx, al", in("dx") PIC2_DATA, in("al") 0x28u8, ...); // slave offset  = 40
```

### ICW3 — Cascade wiring
Written to the **data port**.

```
Master: 0x04  → bit 2 set = IRQ2 is connected to slave
Slave:  0x02  → cascade identity = 2 (I am connected to master IRQ2)
```

In OxideOS (`pic.rs` line 85–102):
```rust
asm!("out dx, al", in("dx") PIC1_DATA, in("al") 0x04u8, ...);
asm!("out dx, al", in("dx") PIC2_DATA, in("al") 0x02u8, ...);
```

### ICW4 — 8086 mode
Written to the **data port**.

```
Value: 0x01
  bit 0 (8086 mode): 1  — use 8086/88 mode (not 8080/85)
  bit 1 (auto EOI):  0  — manual EOI required
```

In OxideOS (`pic.rs` line 104–119):
```rust
asm!("out dx, al", in("dx") PIC1_DATA, in("al") 0x01u8, ...);
asm!("out dx, al", in("dx") PIC2_DATA, in("al") 0x01u8, ...);
```

## Interrupt Mask Register (IMR)

After ICW4, the data ports become the **Interrupt Mask Register**:
- Each bit corresponds to one IRQ (bit 0 = IRQ0, bit 7 = IRQ7)
- **1 = masked (disabled)**, **0 = unmasked (enabled)**

OxideOS starts with all IRQs masked (`0xFF`), then selectively enables them:

```rust
// pic.rs line 121-136: mask everything initially
asm!("out dx, al", in("dx") PIC1_DATA, in("al") 0xFFu8, ...); // all masked
asm!("out dx, al", in("dx") PIC2_DATA, in("al") 0xFFu8, ...);

// pic.rs line 158: then enable cascade (IRQ2 must always be on for slave to work)
unmask_irq(2);
```

`unmask_irq(irq)` reads the current mask, clears the bit for `irq`, writes it back:
```rust
mask &= !(1 << irq_bit);
```

`mask_irq(irq)` sets the bit:
```rust
mask |= 1 << irq_bit;
```

## End of Interrupt (EOI)

**This is the most important runtime detail.** After every interrupt handler runs,
you **must** send an EOI (End of Interrupt) command back to the PIC, or it will
never send another interrupt of that IRQ line again.

```
EOI command = 0x20  (PIC_EOI constant)
```

Rules:
- For IRQ0–IRQ7 (master only): send EOI to master (`0x20`)
- For IRQ8–IRQ15 (slave): send EOI to **both** slave (`0xA0`) **and** master (`0x20`)

In OxideOS (`pic.rs` line 161–179):
```rust
pub unsafe fn send_eoi(irq: u8) {
    if irq >= 8 {
        // Slave PIC must be told first
        asm!("out dx, al", in("dx") PIC2_COMMAND, in("al") PIC_EOI, ...);
    }
    // Master PIC always needs EOI
    asm!("out dx, al", in("dx") PIC1_COMMAND, in("al") PIC_EOI, ...);
}
```

**Symptom of missing EOI:** the first interrupt of that IRQ fires correctly,
but no subsequent ones arrive. The timer stops ticking, or keys only work once.

| IRQ12 | 44 | PS/2 Mouse | `interrupts::init()` → `pic::unmask_irq(12)` |

## Common gotchas

**1. Forgetting `io_wait()` between writes.**
The 8259A is an old ISA chip — very slow. Writing too fast corrupts the init
sequence. The `io_wait()` function writes a dummy byte to port `0x80`
(a POST diagnostic port) to burn ~1 µs of time.

**2. Forgetting to enable IRQ2.**
The slave PIC only works if IRQ2 on the master is unmasked. If you mask IRQ2,
all IRQ8–IRQ15 devices silently stop working.

**3. Wrong EOI order for slave IRQs.**
For IRQ8–15, you must send EOI to the slave *first*, then the master. Sending
to master first causes the master to re-enable before the slave is done.

**4. The PIC is legacy — modern systems use the APIC.**
The 8259A dates from 1976. Modern CPUs have a built-in **Local APIC** and
support an **I/O APIC** for multi-core interrupt routing. OxideOS uses the
8259A because it's simpler and sufficient for single-core QEMU/VirtualBox.
Real production kernels disable the 8259A and use the APIC instead.
