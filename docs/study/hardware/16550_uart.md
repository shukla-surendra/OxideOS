# NS 16550 UART — Serial Port (COM1)

**Source:** `kernel/src/kernel/drivers/serial.rs`

---

## What it is

The **16550 UART** (Universal Asynchronous Receiver/Transmitter) is a serial
communication chip. In OxideOS it serves as the **debug output channel** — all
kernel log messages go here via `SERIAL_PORT.write_str(...)`, and QEMU/VirtualBox
can forward this to a host terminal or file.

Serial communication works by sending one bit at a time over a single wire at a
fixed speed called the **baud rate**. The 16550 handles all the timing, framing
(start/stop bits), and buffering so the CPU just writes bytes to a register.

COM1 is the first serial port, hardwired to base address `0x3F8`.

---

## I/O Ports (base = 0x3F8)

| Offset | Port | When DLAB=0 | When DLAB=1 |
|--------|------|-------------|-------------|
| `+0` | `0x3F8` | Data Register (read = RX, write = TX) | Divisor Latch Low byte |
| `+1` | `0x3F9` | Interrupt Enable Register | Divisor Latch High byte |
| `+2` | `0x3FA` | FIFO Control Register (write) / IIR (read) | — |
| `+3` | `0x3FB` | Line Control Register (includes DLAB bit) | — |
| `+4` | `0x3FC` | Modem Control Register | — |
| `+5` | `0x3FD` | Line Status Register (read-only) | — |

**DLAB (Divisor Latch Access Bit)** is bit 7 of the Line Control Register. Setting it
to 1 remaps offsets `+0` and `+1` from data/IRQ registers to the baud rate divisor.
You write the divisor, then clear DLAB to return to normal operation.

---

## Baud Rate

Baud rate = 115,200 / divisor.

| Divisor | Baud rate |
|---------|-----------|
| 1 | 115,200 |
| 2 | 57,600 |
| 3 | 38,400 ← OxideOS uses this |
| 12 | 9,600 |

OxideOS sets divisor = 3 → **38,400 baud** (`serial.rs` line 38):
```rust
self.outb(LINE_CTRL_REG, 0x80);  // set DLAB=1
self.outb(DATA_REG, 0x03);       // divisor low  = 3
self.outb(INT_ENABLE_REG, 0x00); // divisor high = 0
self.outb(LINE_CTRL_REG, 0x03);  // clear DLAB, set 8N1
```

---

## Line Control Register (LCR) — port 0x3FB

```
Bits [1:0] — Word length
  00 = 5 bits
  01 = 6 bits
  10 = 7 bits
  11 = 8 bits ← OxideOS (value 0x03)

Bit 2 — Stop bits
  0 = 1 stop bit ← OxideOS
  1 = 1.5 or 2 stop bits

Bits [5:3] — Parity
  x00 = No parity ← OxideOS

Bit 7 — DLAB (Divisor Latch Access Bit)
  0 = normal operation ← OxideOS after baud rate set
  1 = access baud rate divisor
```

`0x03` = 8 data bits, 1 stop bit, no parity = **8N1** — the universal serial default.

---

## Line Status Register (LSR) — port 0x3FD

```
Bit 0 (DR — Data Ready):
  1 = a byte has been received and is waiting in the data register

Bit 5 (THRE — Transmit Holding Register Empty):
  1 = the transmit buffer is empty; safe to write another byte
  0 = transmitter is busy; writing would overwrite or be lost
```

OxideOS busy-waits on bit 5 before every write:
```rust
// serial.rs write_byte() — line 65
while (self.inb(LINE_STATUS_REG) & TRANSMIT_EMPTY) == 0 {}
self.outb(DATA_REG, byte);
```

And checks bit 0 before reading:
```rust
// serial.rs read_byte() — line 141
if (self.inb(LINE_STATUS_REG) & DATA_READY) != 0 {
    Some(self.inb(DATA_REG))
} else { None }
```

---

## FIFO Control Register (FCR) — port 0x3FA

```
0xC7 written by OxideOS (serial.rs line 45):

Bit 0 — Enable FIFO:  1 (enable 16-byte hardware FIFO)
Bit 1 — Clear RX FIFO: 1
Bit 2 — Clear TX FIFO: 1
Bits [7:6] — Trigger level: 11 = 14-byte threshold (interrupt when 14 bytes received)
```

Without the FIFO, every received byte would need immediate CPU attention or be lost.
The FIFO buffers up to 16 bytes so the CPU can process them in batches.
OxideOS doesn't use receive interrupts (it polls), but enabling and clearing the FIFO
ensures no stale data from firmware corrupts early reads.

---

## Initialization sequence (`serial.rs init()`)

```
1. Disable interrupts (INT_ENABLE_REG = 0x00)
2. Enable DLAB (LINE_CTRL_REG = 0x80)
3. Set baud divisor low = 0x03, high = 0x00  → 38,400 baud
4. Clear DLAB, set 8N1 (LINE_CTRL_REG = 0x03)
5. Enable+clear FIFO, 14-byte threshold (FIFO_CTRL_REG = 0xC7)
6. Enable RTS/DSR (MODEM_CTRL_REG = 0x0B)
7. Loopback test: write 0xAE, read back — verify chip works
8. Set normal operation (MODEM_CTRL_REG = 0x0F)
```

The loopback test (step 7) briefly enables the UART's internal loopback mode
(`MODEM_CTRL_REG = 0x1E`) — what you write to the TX register immediately appears in
the RX register — letting the driver verify the hardware works before relying on it.

## In OxideOS

The global `SERIAL_PORT` is declared as a `static` and initialized before everything else
in `kmain()` because serial is the first debug output channel:

```rust
// serial.rs line 187
pub static SERIAL_PORT: SerialPort = SerialPort::new(SERIAL_PORT_BASE);
```

Every kernel subsystem calls `SERIAL_PORT.write_str(...)` for debug logging.
In QEMU, launch with `-serial stdio` or `-serial file:serial.log` to capture it.

---

## Common gotchas

**1. Forgetting to clear DLAB before writing data.**
If DLAB is still set to 1, writes to offset `+0` change the baud rate divisor
instead of sending data. Symptom: garbage output at wrong baud rate.

**2. Not waiting for THRE before writing.**
If you write faster than the UART can transmit, bytes get dropped silently.
The busy-wait loop in `write_byte()` prevents this.

**3. Serial is initialized before the heap.**
`SERIAL_PORT` is a static (no allocation needed). It can be used immediately at
kernel entry, before any allocator setup. This is why it's the very first thing
initialized in `kmain()`.
