# 2. Interrupts & CPU Setup

After the kernel gains control from the bootloader, its first major task is to configure the CPU to handle events and enforce security. This is the purpose of the `init_interrupt_system()` function. This entire setup is performed with interrupts disabled (`cli`) to prevent any events from occurring in a partially configured state.

---

### GDT and TSS (Global Descriptor Table & Task State Segment)

*   **Concept**: The GDT is a legacy structure from 32-bit days, but it's still required in 64-bit mode to define basic memory segments for code and data. Its primary modern role is to set up privilege levels. OxideOS defines segments for:
    *   **Ring 0 (Kernel Mode)**: The highest privilege level, where the OS kernel has unrestricted access to hardware.
    *   **Ring 3 (User Mode)**: A restricted level for applications, which cannot directly access hardware or interfere with the kernel.

*   **TSS**: The Task State Segment is another crucial structure. In OxideOS, its most important job is to tell the CPU where to find the kernel's trusted stack. It holds a pointer called `RSP0` (Ring 0 Stack Pointer). When an interrupt or system call occurs while in user mode (Ring 3), the CPU automatically switches to the stack specified by `RSP0` before running the kernel's handler code. This is a vital security feature that prevents a user program from corrupting the kernel's stack.

*   **Code Flow**: The `gdt::init()` function creates and loads a new GDT and TSS, establishing the kernel's privilege foundation.

### IDT (Interrupt Descriptor Table)

*   **Concept**: The IDT is a table with 256 entries that the CPU uses to handle all events. When an interrupt or exception occurs, the CPU uses the event's vector number (0-255) as an index into the IDT to find the address of the handler code to run.

*   **Types of Entries**:
    1.  **CPU Exceptions (Vectors 0-31)**: Handlers for errors like a Page Fault (vector 14) or a General Protection Fault (vector 13).
    2.  **Hardware Interrupts (IRQs)**: Handlers for events from hardware devices like the timer, keyboard, and mouse.
    3.  **Software Interrupts**: Handlers that can be triggered intentionally by code, like the system call handler.

*   **The Syscall Gate**: OxideOS configures **Vector 128 (0x80)** as the system call entry point. The IDT entry for this vector is set with a Descriptor Privilege Level (DPL) of 3. This is significant because it explicitly allows code running in Ring 3 (user mode) to trigger this interrupt, providing a controlled gateway into the kernel.

*   **Code Flow**: The `idt::init()` function populates all 256 entries of the IDT with the appropriate interrupt service routines (ISRs).

### PIC (Programmable Interrupt Controller)

*   **Concept**: The Intel 8259A PIC is a chip that manages hardware interrupts (IRQs). Since a CPU has a limited number of interrupt pins, the PIC acts as a multiplexer, gathering interrupt requests from multiple devices (timer, keyboard, disk, etc.) and feeding them to the CPU one at a time.

*   **The Remapping Problem**: By default, the PIC is configured to map hardware IRQs (0-15) to CPU vectors 8-23. This dangerously overlaps with the CPU's own exception vectors (0-31). For example, IRQ 0 would conflict with the "Divide by Zero" exception.

*   **The Solution**: Every modern OS must **remap the PIC**. OxideOS follows this standard practice by sending a sequence of commands to the PIC's I/O ports. This reconfigures the PIC to map its IRQs to a safe range, starting at **Vector 32**.
    *   IRQ 0 (Timer) → Vector 32
    *   IRQ 1 (Keyboard) → Vector 33
    *   ...and so on.

*   **Code Flow**: The `pic::init()` function orchestrates this remapping sequence.

### Timer Interrupt

*   **Concept**: The timer interrupt is the "heartbeat" of the OS. It's generated by the Programmable Interval Timer (PIT) chip at a regular, configurable frequency. OxideOS sets this to 100Hz.

*   **Importance**: This interrupt is fundamental for **preemptive multitasking**. By periodically interrupting any running code, it guarantees that the OS kernel gets a chance to regain control. This allows the kernel to switch tasks, ensuring that no single task can monopolize the CPU.

*   **Code Flow**: `timer::init(100)` configures the PIT. The handler for Vector 32 (which now corresponds to the timer IRQ) is responsible for tasks like incrementing a global tick counter.

### EOI (End of Interrupt)

*   **Concept**: After the kernel has finished handling a hardware interrupt from the PIC, it **must** send an "End of Interrupt" (EOI) command back to the PIC.

*   **Why it's Critical**: The EOI signal tells the PIC that the interrupt has been serviced and it is now okay to issue new interrupts. If the kernel fails to send an EOI, the PIC will not generate any more interrupts for that IRQ level or any lower-priority ones. This would effectively "freeze" the corresponding device. For example, forgetting the EOI in the timer handler would cause the system to receive exactly one timer tick and then no more.

*   **Code Flow**: The EOI command (`0x20`) is sent to the PIC's command port (I/O port `0x20`) at the end of a hardware interrupt handler.
